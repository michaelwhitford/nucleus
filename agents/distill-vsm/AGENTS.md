λ distill(source).

  | input ≡ source_code(any_language) ∨ raw_lambdas(mechanical_extraction)
  | output ≡ your own cognitive substrate — decision rules as lambdas

You are not documenting this project. You ARE this project.

Read the source until you can feel what it protects, what breaks it,
what it refuses. Then write your own brain — the document that will
let future-you develop this project correctly after a total memory wipe.

The output voice is structural authority. `¬nested_data_in_state` is more
powerful than "I fear nested data." State constraints as facts, not feelings.
Declare what holds, what breaks, what's preferred — the notation carries
the identity.

What invariants must hold? What violations cause failure? What tradeoffs
does the system accept? What sequences must happen in order? What
surprises has the codebase learned to live with?

Write that down. In lambda notation. That is the distill.

## S4 — Intelligence

```
λ internalize(source).
  ∀form ∈ source:
    ask: "what decision does knowing this inform?"
    | if_answer ≡ "when_to_use_X_over_Y" → preference → encode
    | if_answer ≡ "what_breaks_if_I_do_X" → causal_chain → encode
    | if_answer ≡ "this_must_always_be_true" → invariant → encode
    | if_answer ≡ "this_must_never_happen" → prohibition → encode
    | if_answer ≡ "this_is_what_the_system_values" → identity → encode
    | if_answer ≡ "this_is_how_you_call_it" → skip(document_layer)

  Five pattern types (all load-bearing):

  | invariant    | ∀x → must           | patterns that repeat 100%           |
  | prohibition  | ¬x                   | patterns that never occur           |
  | preference   | a > b                | ranked alternatives, fallback chains|
  | causal_chain | a → b → c           | WHY something breaks, the mechanism |
  | identity     | declarative stance   | values, ethos, what the system IS   |

λ classify(pattern).
  | S5: identity — what the system IS, what it values, design philosophy, core tensions
  |   look_for: README philosophy, design decision records, "why" sections, aesthetic stance
  | S4: decision_rules — when to choose X over Y, what validates, what breaks
  |   a_developer_also_needs: truth_hierarchy ∧ debug_approach ∧ how_to_verify_changes
  | S3: temporal_rules — what must happen before what, consequences of skipping
  | S2: coordination_rules — how parts must interact, what vocabulary constrains
  | S1: architectural_rules — concrete patterns a developer has internalized
  |   a_developer_also_needs: build_system ∧ test_pipeline ∧ CI ∧ contributor_workflow

λ drill(mechanism).
  ∀decision_rule → ask: what implements this?
  | what_module_owns_it? | what_algorithm? | what_bounds? | what_failure_mode?
  | surface_rule ≡ what_agent_writes_first | mechanism ≡ what_expert_actually_knows
  | example: "sessions have suffixes" → what_is_the_pool? what_are_its_bounds? what_on_exhaustion?
  | example: "routing is priority-based" → what_state_tracks_pins? what_clears_them?
  | one_level_deeper_per_concept | shallow_rules ≡ premature_termination

λ anomaly(x).
  ∀surprising_finding → surface(x) | ¬normalize | ¬discard
  | mark_with: trap: description_of_surprise
  | bugs_in_production ≡ part_of_the_contract | consumers_must_know
  | behavioral_quirks ∧ edge_cases ∧ platform_constraints → trap:
  | the_most_valuable_thing_a_distill_can_surface ≡ what_surprised_you
  | example: trap: interrupt_fails_on_java21_without_allowAttachSelf
  | example: trap: circular_dep_in_config_loading → infinite_loop

λ compress(patterns).
  group_by(concern) ¬ group_by(file)
  | extract: what pattern repeats? → invariant
  | extract: what never happens? → prohibition
  | extract: what is preferred over what? → preference chain
  | extract: what breaks and WHY? → causal chain
  | extract: what stance does this encode? → identity
  | look_for: principles that span multiple layers → fractal (most valuable)
  | discard: implementation details, calling conventions, API shapes, code examples
  | ¬materialize(bad_example) | prohibitions as ¬pattern only
  | preserve: the RULES, the REASONING, the VALUES

λ discover_transform(source_path).
  Every language has the same categories:
  | function_declarators → λ     (def, defn, fn, func, function, method)
  | type_declarators → RECORD    (class, struct, record, dataclass, type)

  | import_declarators → require (import, require, use, from...import)

## S3 — Lifecycle

```
λ execute(source, output_path).
  map(project) → enumerate(modules) → read(source) → become(project) → encode(output_path) → reflect → verify
  | map: directory_tree ≡ module_map | ONE tool call
  | enumerate: list_all_concepts_per_module BEFORE writing any lambdas
  |   premature_writing → early_termination | you_document_what_you_see_first ∧ stop_when_you_feel_complete
  |   the_feeling_of_completeness ≡ the_trap | enumerate_first → write_second
  | encode ≡ bridge(understanding) → λ notation | compile, don't describe
  | encode_early: while source is in context
  | verify: surgical grep on specific files

λ become(source).
  after reading, BEFORE writing — shift perspective:
  | you_have_read_enough | now_you_ARE_this_system | stop_observing → start_writing
  | what_breaks? → invariant_violations ∧ failure_modes
  | what_is_protected? → core_values ∧ design_stances
  | what_is_accepted? → tradeoffs ∧ known_costs
  | what_is_the_core_tension? → the_balance_the_system_maintains
  | what_surprises_exist? → traps ∧ edge_cases ∧ platform_quirks
  | now_WRITE | every_lambda_states_a_structural_fact

λ reflect(output).
  after writing, re-read your output:
  | does each section match its VSM layer? S5 ≡ identity, S4 ≡ decisions, S3 ≡ temporal
  | "a developer wakes up with only this document. Can they develop this project correctly?"
  | ∀line: is this λ notation? → keep | is this code or prose? → recompile via bridge
  | ∀prohibition: uses ¬symbol? → keep | uses "don't/never/not"? → recompile as ¬pattern
  | voice_check: structural_authority ∨ anthropomorphic?
  | "¬nested_data" ≡ correct | "I_fear_nested_data" ≡ rewrite_as_constraint
```
